1. Three-Layer Architecture

The project is structured around three independently operating layers connected via communication protocols. This week, a parallel Commentary Middleware module was added alongside Layer 2:

Layer 3
AI Decision Layer IBM Granite (LLM) / Python AI logic — Receives vehicle state data and outputs driving decisions (steering, throttle, brake)
↕ UDP Socket
Layer 2
Communication Middleware (Driving Control) Python scripts (UDP Socket) — Bridges game engine and AI model; handles data format conversion and control command forwarding
↕ UDP :3101 / CSV
New NEW
Commentary Middleware (Race Commentary) FastAPI + WebSocket + Context Manager — Reads telemetry data, drives AI-generated live race commentary, displayed via Web UI
↕ Physics Engine / Sensors
Layer 1
Game Engine Layer ✅ TORCS (C++ source) + SCR Patch — Physics simulation, track rendering, vehicle sensor data generation broadcast via UDP

2. Overall System Logic

2.1 Driving Control Loop (Existing)

1
TORCS physics engine calculates vehicle state per frame (speed, position, distance to track boundaries, etc.)
2
Built-in scr_server driver broadcasts state data over UDP and waits for external control commands
3
Python middleware listens on the port, formats received data, and sends it to IBM Granite for decision-making
4
AI returns decisions (steering angle, throttle, brake); Python sends commands back to TORCS via UDP
5
scr_server applies commands, completing one "Perception → Decision → Action" loop

2.2 Commentary Data Flow (New This Week) NEW

1
TORCS human driver module pushes telemetry (speed, race position, damage, etc.) as CSV rows via UDP :3101 each frame
2
Commentary middleware background thread receives UDP packets and parses them into structured telemetry dictionaries
3
Context Manager filters fields, converts to natural language, adds to history, and trims to token budget
4
Calls AI API (OpenAI / Anthropic / Ollama) to generate commentary; receives streaming response
5
WebSocket streams tokens to the browser UI in real time, rendering the commentary as a chat-style conversation

3. Layer 1: Game Engine Setup (Completed Last Week)

3.1 Environment Selection

  • Initial download was the Windows pre-compiled binary (no source code) — unsuitable for modification
  • Switched to the official SourceForge source tarball (.tar.bz2), which includes src/, configure, and Makefile.in

3.2 Build Environment: WSL2 + Ubuntu

  • Native Windows compilation of TORCS requires legacy VS versions (VS 6 / VS 2008) — impractical
  • Installed WSL2 + Ubuntu with VS Code WSL extension: "edit on Windows, build and run on Linux"

3.3 SCR Patch Source

  • Stock TORCS has no external AI communication support — the SCR (Simulated Car Racing) Patch adds it
  • Used the pre-patched repository: github.com/fmirus/torcs-1.3.7 (includes scr_server module for UDP communication)

3.4 Build Commands

export CFLAGS="-fPIC"; export CPPFLAGS=$CFLAGS; export CXXFLAGS=$CFLAGS

4. Commentary Middleware Development (Completed This Week) NEW

4.1 Design Goals

Build an independent commentary system that taps into TORCS telemetry without interfering with the driving control pipeline. The system uses a large language model to generate real-time race commentary, displays it through a visual web interface, and supports flexible switching between AI providers (OpenAI / Anthropic / local Ollama).

4.2 SillyTavern-Inspired Context Management

The core architecture is modeled after SillyTavern's prompt management strategy:

SillyTavern FeatureCommentary Middleware Implementation
Context Size Limitmax_context_tokens slider — controls the total token budget sent to the AI
Trim StrategyTrims oldest or newest messages to keep history within budget
System Prompt / Character CardCommentator persona text field — customizable commentary style and focus
Keep in Context (pin)Message.pinned flag — pinned messages are never trimmed
World Info / Author's Noteformat_telemetry() converts raw telemetry into a natural language description block
Token CounterReal-time progress bar in Web UI showing used / total tokens

4.3 File Structure

├── server.py           # FastAPI server (REST + WebSocket + UDP listener thread)
├── context_manager.py  # Context window management (token budget, trimming, formatting)
├── requirements.txt    # fastapi, uvicorn, httpx
└── static/
    └── index.html      # Web UI (dark theme, SillyTavern-style layout)

4.4 Key Features

  • UDP Real-Time Ingestion: Background thread listens on :3101 and parses each frame's TORCS CSV row into a structured telemetry dictionary
  • CSV File Replay: Supports loading historical CSV files and triggering commentary from the latest data point
  • AI API Configuration: UI panel allows switching provider, entering Base URL / API Key / model name — changes apply immediately
  • Automatic Commentary: Configurable timer (e.g., every 10 seconds) reads latest telemetry and triggers commentary automatically
  • Streaming Output: AI responses are streamed token-by-token via WebSocket for a real-time typewriter effect in the browser
  • Field Filtering: Chip-based UI to select which telemetry fields are included in the AI prompt, controlling token usage
